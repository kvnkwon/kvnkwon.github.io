SQL vs NoSQL

SQL vs NoSQL translates to basically relational versus non-relational databases. Some examples of relational databases include: Oracle, MySQL, and SQL Server. Non-relational database examples include: MongoDB, CouchDB, and more.

What is NoSQL?

NoSQL encompasses a variety of different database technologies and was created in response to a rise in the volume of data stored about users, objects and products, the frequency in which this data is accessed, and performance and processing needs.

Facebook, Twitter, and Digg's increasing growth prompted a switch from SQL to NoSQL databases. They found that NoSQL databases were more scalable, reliable, and easier to manage.

The argument against SQL (Common arguments though not 100% proven true):

  • Joins in relational databases can slow the system down significantly, especially when millions of users are doing lookups against tables with millions of rows of datas. Google and Amazon found this to be especially true so they ended up creating their own non-relational systems.
  • Relational data does not map well to typical programming structures that often consist of complex data types or hierarchical data. Data such as XML is especially difficult because of its hierarchical nature. Complex objects that contain objects and lists inside of them do not always map directly to a single row in a single table.
  • Relational data does not map well. Combine that with the need to handle the syntax of SQL, and writing client code for accessing SQL databases become difficult.
  • Basically relational databases were not designed to cope with the scale and agility challenges that face modern applications, nor were they built to take advantage of the cheap storage and processing power available today.

How valid are these claims? Huge banks that serve thousands of branches and millions of customers, including hundreds of thousands of online members, survive just fine using SQL. This leads us to...

The argument for SQL:

  • You're not as big as Google and you're not searching as much data as Google. Further today's relational systems such as Oracle are extremely powerful and mature and can handle joins at the performance level that most systems require.
  • Most data can be mapped to a relational database, even XML and complex objects.
  • Code libraries aid in the development of client code, aren't that difficult to use.
